# ADR-018: Sentinel Learning Loop — Compose from Best-in-Class

**Status:** Accepted
**Date:** 2026-04-01
**Deciders:** Chris (Batman), Alfred (S61)

## Context

MISSION.md (S60) defines Sentinel as the Batcave's learning engine:
observe user behavior → discover patterns → propose automations →
Robin executes → Sentinel measures effectiveness → loop.

Current Sentinel (1,394 lines) is a system health monitor only. None of
the five learning-loop stages are implemented. The gap between vision and
reality is the entire middle of the pipeline: pattern discovery, proposal
generation, and effectiveness measurement.

Build-vs-Buy Gate (ADR-005) requires evaluating existing tools first.

## Decision

**Compose the learning loop from best-in-class open-source tools.**
Write only ~200 lines of glue code. No custom observation engine.

### Intelligence Hierarchy (S61 correction)

Proposals are generated by the **most intelligent actor available**:
1. **Alfred** (Claude) — when a Cowork/CLI session is active
2. **Robin** (Ollama) — when Alfred is offline (Sole Digital Survivor mode)

Sentinel observes; Alfred/Robin think. Sentinel never generates proposals
itself — it collects signal and feeds it to the intelligence layer.

## Architecture

```
OBSERVE          ActivityWatch (local, privacy-first)
    ↓            + Sentinel's existing system observations
    ↓
DISCOVER         PM4Py (process mining) + PrefixSpan (sequential patterns)
    ↓            Runs as a Sentinel scan phase, every 6 hours
    ↓
PROPOSE          Alfred (Claude) when available → Robin (Ollama) as fallback
    ↓            Patterns → structured prompt → proposal JSON
    ↓
EXECUTE          Robin's existing InitiativeEngine + DirectiveTracker
    ↓            Proposals written to robin-inbox/
    ↓
MEASURE          Before/after comparison via ActivityWatch API
                 Did user stop doing it manually? → refine or retire
```

### Components

| Component | Source | Custom Code |
|-----------|--------|-------------|
| Window/app/browser tracking | ActivityWatch | 0 lines (install only) |
| Event log adapter | Custom | ~50 lines — query AW API, format for PM4Py |
| Pattern discovery | PM4Py + PrefixSpan | ~40 lines — run miners on event log |
| Proposal generation | Alfred/Ollama | ~80 lines — format prompt, parse response |
| Robin intake bridge | Custom | ~30 lines — write to robin-inbox/ |
| Effectiveness measurement | Custom | ~30 lines — before/after AW comparison |

**Total custom code: ~230 lines.** Everything else is maintained upstream.

## Options Considered

### Option A: Build Custom (Rejected)
| Dimension | Assessment |
|-----------|------------|
| Complexity | High — keyloggers, window hooks, event databases |
| Maintenance | High — Windows API changes, privacy compliance |
| Time to value | Weeks |

### Option B: Compose from OSS (Accepted)
| Dimension | Assessment |
|-----------|------------|
| Complexity | Low — glue code only |
| Maintenance | Low — upstream maintains heavy components |
| Time to value | Hours for prototype |

## Trade-offs

- **Easier:** Rapid prototype, upstream security patches, tested at scale
- **Harder:** Dependent on ActivityWatch stability, API may change
- **Revisit:** If AW is too heavy for Oracle's resources, consider lighter
  alternatives (pure Windows Activity History API parsing)

## Consequences

1. ActivityWatch must be installed on Oracle (winget or direct download)
2. `pip install pm4py prefixspan` on Oracle's Python 3.12
3. Sentinel gains a new scan phase: `_scan_behavioral_patterns()`
4. New module: `rudy/sentinel_learning.py` (~230 lines of glue)
5. Robin's inbox gains a new message type: `automation-proposal`
