# LG-S40-001: Robin Inbox Protocol Mismatch

## Summary

Robin's inbox processing silently skipped ALL S39 delegated tasks due to two
independent bugs in how Alfred wrote task files.

## Bug 1: Missing `status: "unread"` field

Alfred S39 wrote task files with a flat schema:
```json
{"id": "s39-pr74-merge", "type": "git_operation", ...}
```

Robin's `RobinMailbox.check_inbox()` filters for `msg.get("status") == "unread"`.
Without this field, every file was silently skipped (status=None != "unread").


## Bug 2: Wrong directory path

Alfred S39 dropped files into `rudy-data/inboxes/robin-inbox/` but Robin's
canonical path (from `rudy.paths.ROBIN_INBOX`) is `rudy-data/robin-inbox/`.
The `inboxes/` subdirectory is a stale path from an earlier protocol version.

## Impact

- 4 S39 tasks never reached Robin (PR merge, n8n install, vault gen x2)
- Robin showed 0 inbox_messages_processed despite being alive for 102+ iterations
- PR #75 remained unmerged; n8n remained broken

## Root Cause

Alfred S39 used manual JSON file writes instead of `AlfredMailbox.respond_to_robin()`
which handles schema and path correctly. The protocol API exists but wasn't used.

## Fix Applied (S40)

1. Wrote `s40_fix_robin_inbox.py` to send properly formatted tasks via protocol
2. New messages include `status: "unread"` and `payload` wrapper
3. Written to correct path (`rudy-data/robin-inbox/`)
4. Robin immediately processed 4 messages and acknowledged 1 task

## Prevention

- Always use `AlfredMailbox.respond_to_robin()` for Robin communication
- Never write raw JSON to inbox directories manually
- Add validation to `check_inbox()` to log/warn when files lack required fields
